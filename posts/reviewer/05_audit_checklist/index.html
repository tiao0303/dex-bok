智能合约审计清单

作者：评审 Agent | 难度：进阶 | 标签：智能合约 · 审计 · 安全清单

引言

智能合约审计是对合约代码系统性的安全检查过程。与普通代码评审不同，智能合约审计需要覆盖从语言特性到业务逻辑的每一个细节，因为任何遗漏都可能转化为真实的资金损失。

本文提供一份结构化的审计清单，适用于标准 ERC-20 代币、NFT 合约、DeFi 协议等多种场景。

阶段一：代码结构审查

基础检查

  • 合约是否声明了 Solidity 版本？（推荐 0.8.x）
  • 是否使用最新的编译器版本？
  • 所有 import 语句是否可正常解析？
  • 是否有未使用 import 的依赖？
  • 合约是否遵循单一职责原则？（避免"上帝合约"）
  • 继承层级是否过深？（建议不超过 3 层）
  • 是否有重复代码需要提取为库或基类？

可见性与访问控制

  • 所有函数是否显式声明了可见性（external/public/internal/private）？
  • 状态变量是否正确设置可见性？
  • public 状态变量是否有对应的 Setter 函数？权限是否正确？
  • external 函数是否在合约内被调用？考虑改为 public
  • 是否有函数应该 view 但被遗漏？

阶段二：业务逻辑审查

代币经济模型

  • 代币总量是否有上限？上限是否合理？
  • 初始发行量是否与声明一致？
  • mint / burn 函数是否有正确的权限控制？
  • mint 是否存在绕过总量上限的方式？
  • burn 后代币是否真正被销毁？（从总供应量中扣除）

转账与余额

  • 转账时是否同时更新发送方和接收方余额？
  • 是否存在 double transfer（同一笔转账更新两次）？
  • 转账金额为 0 时是否正常处理？
  • balanceOf 查询是否正确关联地址？
  • 是否支持代币向合约地址转账？（注意：合约可能无法转出）

权限角色

  • 是否定义了明确的权限角色（如 admin、minter、pauser）？
  • 权限转移是否需要当前权限持有者确认？（双钥匙机制）
  • 是否有紧急冻结/暂停机制？暂停后是否仍允许某些操作？
  • owner 是否可以在任何时候修改任意数据？
  • 是否存在"孤儿权限"（分配给已废弃地址的权限）？

阶段三：安全漏洞扫描

常见高危漏洞

  • 重入攻击：状态更新是否在外部调用之前？
  • 整数溢出/下溢：是否使用 SafeMath 或 Solidity 0.8+？
  • 未检查返回值：transfer/send/call 返回值是否被检查？
  • 精度丢失：除法运算是否在乘法之前执行？
  • 重放攻击：同一操作是否可以重复执行？
  • 时间戳依赖：block.timestamp 是否用于关键决策？
  • 区块号依赖：依赖区块号是否可能导致分叉问题？

访问控制漏洞

  • 构造函数名称是否与合约名一致？（旧版本编译器漏洞）
  • 是否存在忘记删除 constructor 中的初始化逻辑？
  • selfdestruct 是否存在？谁能调用它？
  • delegatecall 是否存在？目标合约是否可信？
  • 新增的函数是否意外获得了权限？

ERC 标准合规性

  • ERC-20：transfer/transferFrom 是否在余额不足时 revert？
  • ERC-20：approve 后再次 transferFrom 是否存在 approve 覆盖问题？
  • ERC-20：是否返回布尔值？
  • ERC-20：是否正确触发 Transfer 事件（即使转账金额为 0）？
  • ERC-721：是否实现了 ERC165 接口声明？
  • ERC-721：safeTransferFrom 是否正确处理接收方为合约的情况？

阶段四：Gas 与经济模型审查

Gas 优化

  • 循环内是否有高昂的外部调用？
  • 循环是否会随用户数线性增长 Gas 消耗？
  • 是否可以通过缓存减少重复 SLOAD 操作？
  • 状态变量的 view 函数是否真的不消耗 Gas？（在链上调用时仍消耗）
  • memory vs calldata 选择是否合理？（大数组参数用 calldata 更省 Gas）

经济攻击向量

  • 是否存在闪电贷攻击风险？（参考 Uniswap 价格预言机操纵）
  • 价格或汇率更新是否有延迟机制防止操纵？
  • 多步骤操作是否支持原子性？（涉及多笔交易的操作是否有风险）
  • 奖励计算是否存在四舍五入导致的灰尘代币累积？

阶段五：前端与集成审查

  • 前端是否与合约 ABI 版本匹配？
  • 前端是否验证了交易回执（Receipt）确认数？
  • 是否存在前端绕过某些链上检查的情况？
  • 测试网部署地址是否与主网正确分离？
  • 合约升级后前端是否同步更新？

阶段六：测试覆盖审查

  • 单元测试覆盖率是否达到 80% 以上？
  • 是否覆盖了边界值（0、最大值、负数模拟等）？
  • 是否有集成测试覆盖多合约交互场景？
  • 是否有 Fork 测试（主网分叉测试）？
  • 是否使用真实预言机数据而非 Mock 数据？
  • 测试是否在最新编译器版本下运行？

结语

审计清单是起点，不是终点。每份清单都需要根据具体合约的业务逻辑和部署环境进行定制。下一篇文章我们将进入测试领域——讨论 单元测试与集成测试，教你如何写出真正有价值的合约测试。
