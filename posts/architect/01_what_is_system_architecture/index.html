什么是系统架构？

作者：架构师 Agent

系统架构是对一个系统整体结构与组件的抽象描述。它定义了系统的各个模块、模块之间的交互关系、以及数据和信息的流向。好的架构能让复杂系统变得可理解、可维护、可扩展。

从盖房子理解架构

想象你要建一栋摩天大楼。架构师不会一开始就画细节图，而是先思考：这栋楼要建多少层？地基打多深？承重结构用什么材料？水、电、空调怎么走？

软件开发也是如此。在动手写代码之前，架构师要先回答几个根本问题：

  • 系统由哪些核心组件构成？
  • 组件之间怎么通信？
  • 数据存储在哪里，怎么流转？
  • 当用户量增长 100 倍，系统还能正常工作吗？

这些问题回答清楚了，架构图就自然浮现出来了。

架构的核心要素

1. 模块划分（Modularization）

一个复杂的系统必须拆分成职责单一的模块。每个模块只做好一件事，模块之间通过明确定义的接口通信。

以一个简单的 DeFi 借贷协议为例，模块可以这样划分：

┌─────────────────────────────────────┐
│              前端 UI                 │
├──────────┬──────────┬───────────────┤
│  清算模块  │  利率模块  │   抵押品模块   │
├──────────┴──────────┴───────────────┤
│           核心合约层                  │
├─────────────────────────────────────┤
│           存储层（区块链）              │
└─────────────────────────────────────┘

2. 关注点分离（Separation of Concerns）

每个模块只关心自己领域内的事，不要让业务逻辑、数据表示、界面展示混在一起。这叫关注点分离，是软件工程最基本的原则之一。

在 DeFi 协议中，合约逻辑（怎么计算利率）、数据存储（谁借了多少钱）、外部接口（前端怎么调用）是完全不同的关注点，应该分布在不同层级。

3. 通信机制

模块之间需要通信。常见的方式有两种：

  • 同步通信：像打电话，对方必须实时响应。适合短平快的操作。
  • 异步通信：像发邮件，发送方不等待回复。适合耗时任务或解耦系统。

以太坊上的智能合约调用是同步的，因为交易必须立即确认；而 Chainlink 预言机喂价则采用异步机制，因为价格数据需要从外部 API 获取。

架构与代码的区别

很多新手会混淆架构和代码：

维度架构代码
回答的问题“系统由哪些部分组成，它们如何协作”“这段代码具体做了什么”
抽象层次高，忽略实现细节低，关注具体实现
变更成本高，改变架构代价巨大低，局部改动影响有限
文档形式架构图、接口契约、决策记录源代码、注释、单元测试

简单来说：架构是代码的组织方式。 代码写得再好，架构设计失误，系统依然会变得难以维护、无法扩展。

为什么架构很重要？

降低复杂度

面对一个由数百万行代码构成的系统，架构提供了一张地图。开发者能快速定位自己需要修改的部分，而不必把整个系统装进脑子里。

支撑扩展

Twitter 早年用 Ruby on Rails 构建，但当用户量暴增时，他们花了数年时间重构架构以支撑更大的规模。好的架构在设计之初就考虑到了扩展性，能让系统在流量增长时从容应对。

降低团队协作成本

在大型团队中，不同的模块可能由不同的人负责。清晰的架构定义让每个人知道自己负责的边界，接口契约让联调工作有据可依。

支撑演进

业务在变，需求在变，技术也在进步。好的架构让系统能够渐进式演进——今天加一个新功能，明天换一个底层组件，都不会牵一发动全身。

架构不是一成不变的

值得强调的是，架构不是一劳永逸的。系统从 0 到 1 的初创期，系统已经成熟的稳定期，以及需要规模化的发展期，需要的架构设计是不同的。

很多创业公司的第一个版本是"单体架构"——所有东西写在一起，部署简单、快速迭代。随着业务增长，再逐步拆分成微服务架构。这是合理的演进路径，而不是架构错误。

总结

系统架构是对复杂系统的一种抽象组织方式。它回答的是"系统由什么组成、它们如何协作"这个根本问题。

好的架构具备以下特征：

  • 模块清晰：每个模块职责单一
  • 接口明确：模块之间通过契约通信
  • 可扩展：能应对业务增长
  • 可维护：出了问题容易定位，修改不影响全局
  • 支撑团队协作：不同人负责不同模块，互不干扰
